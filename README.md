## FamilyTree — дипломный проект

FamilyTree — это веб‑приложение для создания и визуализации семейного древа с поддержкой мультимедиа, совместной работы членов семьи и хранения воспоминаний. Проект состоит из backend‑части на Django/DRF и frontend‑части на Next.js (App Router), упакованных в Docker и проксируемых через Nginx.

---

### Стек технологий

- **Backend**
  - `Python`, `Django 5`, `Django REST Framework`
  - `PostgreSQL` в качестве основной базы данных
  - `djangorestframework-simplejwt` — аутентификация через JWT (эндпоинты `/api/auth/login/`, `/api/auth/refresh/`)
  - `django-cors-headers` — CORS между фронтом и бэком
  - Модели: `UserProfile`, `Family`, `TreeNode`, `TreeEdge`, `Memory`, `*Media`

- **Frontend**
  - `Next.js` (App Router, TypeScript)
  - `React`, `@tanstack/react-query` — работа с API и кеширование данных
  - `Zustand` — хранение сессии пользователя (`useAuthStore`)
  - `Tailwind CSS` — стилизация интерфейса
  - UI‑компоненты (кнопки, инпуты, диалоги) и иконки `lucide-react`
  - `sonner` — toast‑уведомления

- **Инфраструктура**
  - `Docker` и `docker-compose` — оркестрация сервисов (`db`, `backend`, `frontend`, `nginx`)
  - `nginx` — статический сервер и reverse‑proxy

### Структура проекта

- `backend/` — Django‑проект `familytree`
  - `familytree/settings.py` — настройки Django, база данных, JWT, CORS
  - `familytree/urls.py` — API‑роуты (`/api/...`, `/api/auth/...`)
  - `api/models.py` — доменные модели:
    - `UserProfile` — расширение стандартного пользователя (даты рождения и смерти, вычисление возраста)
    - `Family` — семейные группы
    - `TreeNode` / `TreeEdge` — узлы и связи на древе
    - `Memory`, `MemoryMedia`, `NodeMedia`, `FamilyMedia` — воспоминания и медиафайлы
  - `api/serializers.py` — DRF‑сериализаторы с валидацией и подготовкой данных для фронта
  - `api/views.py` — viewset’ы и функции:
    - `RegisterViewSet` (`/api/auth/register/`)
    - `me` (`/api/auth/me/`) — профиль текущего пользователя
    - `FamilyViewSet`, `TreeNodeViewSet`, `TreeEdgeViewSet`, `MemoryViewSet`

- `frontend/` — Next.js‑приложение
  - `app/` — страницы:
    - `(auth)/login` и `(auth)/register` — вход и регистрация
    - `/dashboard` — основной дашборд
    - `/dashboard/profile` — профиль пользователя и редактирование данных
    - `/dashboard/family/[id]` и вложенные страницы (дерево, воспоминания и т.д.)
  - `components/` — переиспользуемые компоненты:
    - `Navbar`, `Footer` — общая навигация и футер
    - `NodeEditor`, `NodeViewer` — работа с узлами дерева
    - `MemoryViewer`, `EditMemoryModal`, `FamilyCard` и др.
  - `lib/api.ts` — тонкая обёртка над `fetch` с:
    - базовым URL `NEXT_PUBLIC_API_URL`
    - таймаутом запросов
    - обработкой ошибок и автоматическим выходом при истёкшей сессии (401)
  - `store/useAuthStore.ts` — Zustand‑хранилище пользователя и токена (persist в `localStorage`)

### Аутентификация и работа с сессией

- **Регистрация** (`/register` → `POST /api/auth/register/`)
  - Валидация логина, email и пароля на фронте через `lib/security.ts`
  - При успешной регистрации backend возвращает `access`/`refresh` токены и данные пользователя
  - Frontend сохраняет токен в `localStorage`, кладёт пользователя в `useAuthStore` и перенаправляет в `/dashboard`
  - Ошибки (например, занятый логин/email) отображаются в toast’ах и под формой

- **Логин** (`/login` → `POST /api/auth/login/`)
  - Используется эндпоинт SimpleJWT `TokenObtainPairView`
  - При успешном входе:
    - `access` токен сохраняется в `localStorage` и сторе
    - выполняется запрос `GET /api/auth/me/` для получения профиля
  - При ошибке (неверные данные/401):
    - **страница не перезагружается**, токен очищается
    - пользователь видит понятное сообщение об ошибке (toast + подпись под формой)

- **Проверка сессии**
  - Все методы `api.get/post/patch/delete/uploadFile` автоматически подставляют `Authorization: Bearer <token>`
  - При 401 (истёкший/невалидный токен, кроме авторизационных эндпоинтов) токен удаляется и выполняется redirect на `/login`
  - В `Navbar` и `Footer` состояние авторизации берётся из `useAuthStore`, поэтому:
    - авторизованным пользователям показываются ссылки на дашборд/профиль
    - **ссылка "Войти" в футере скрыта для уже вошедших пользователей**

### Работа с датами (рождение/смерть и воспоминания)

#### Backend‑валидация

Для `UserProfile` и `TreeNode` реализована строгая логическая валидация дат:

- Нельзя указать дату рождения в будущем
- Нельзя указать дату смерти в будущем
- Нельзя указать дату смерти раньше даты рождения

Это реализовано:

- В `api/models.py`:
  - метод `clean()` в `UserProfile` и `TreeNode` выбрасывает `ValidationError` при нелогичных датах
- В `api/serializers.py`:
  - `UserSerializer.update()` дополнительно проверяет `birth_date`/`death_date` перед сохранением профиля
  - `TreeNodeSerializer.validate()` валидирует даты при создании/редактировании узлов дерева

#### Frontend‑валидация

На фронте добавлены пользовательские проверки и ограничения:

- **Редактор узла дерева** (`components/NodeEditor.tsx`)
  - Обязательные поля: имя и дата рождения
  - При сохранении проверяется:
    - рождение не в будущем
    - смерть не в будущем
    - смерть не раньше рождения
  - Поля дат имеют HTML‑ограничения:
    - `birthDate`: `max = сегодня`
    - `deathDate`: `min = birthDate`, `max = сегодня`

- **Профиль пользователя** (`app/dashboard/profile/page.tsx`)
  - Аналогичная логика валидации на клиенте перед отправкой
  - Поля дат имеют `max = сегодня`, `deathDate` также имеет `min = birthDate`
  - При ошибках пользователь видит toast и сообщение под формой

- **Воспоминания** (`Memories`)
  - В формах создания и редактирования воспоминаний дата ограничена `max = сегодня`, чтобы нельзя было создать событие в будущем (с точки зрения интерфейса).

### Генеалогическое древо: связи и поколения

Связи в дереве трактуются строго как **родитель → ребёнок**: у каждого ребра узел `source` — родитель, узел `target` — ребёнок. Это обеспечивает логичное древо без «перепутанных» связей (сын–мать, бабушка–внук одной дугой и т.п.).

#### Правила для связи (родитель → ребёнок)

- У **обоих** узлов должна быть указана дата рождения; иначе связь создать нельзя.
- **Родитель должен быть рождён раньше ребёнка**: дата рождения родителя строго раньше даты рождения ребёнка. Разница в возрасте не проверяется — важно только соблюдение порядка по датам.

Проверки выполняются:

- **При создании связи** (`POST /api/edges/`): в `api/views.py` в `TreeEdgeViewSet.perform_create` вызывается `validate_parent_child_edge(source.birth_date, target.birth_date)` из `api/utils.py` — проверяется только то, что родитель рождён раньше ребёнка.
- **При изменении даты рождения узла**: в `api/serializers.py` в `TreeNodeSerializer.validate` при обновлении узла проверяются все его исходящие и входящие рёбра; новая дата рождения не должна нарушать правило «родитель рождён раньше ребёнка» ни для одного из этих рёбер.

#### Поведение на фронте

- На странице дерева при создании связи (перетаскивание от одного узла к другому) порядок **родитель → ребёнок** подставляется автоматически по датам рождения: если пользователь тянет от младшего к старшему, запрос уходит уже в правильном порядке (старший = родитель, младший = ребёнок). Под заголовком отображается подсказка: «Связь: от родителя к ребёнку (можно тянуть в любую сторону)».

### Медиафайлы и Git

- Файлы, загружаемые пользователями (фото и видео семей, узлов, воспоминаний), хранятся в каталоге **`backend/media/`** и **не коммитятся** в репозиторий.
- Чтобы при клонировании репозитория структура папок под медиа сохранялась, в репозитории лежат только пустые директории с файлом-заглушкой **`.gitkeep`**:
  - `backend/media/.gitkeep`
  - `backend/media/families/.gitkeep`, `backend/media/nodes/.gitkeep`, `backend/media/memories/.gitkeep`
  - `backend/media/families/media/.gitkeep`, `backend/media/nodes/media/.gitkeep`
- Игнорирование содержимого медиа задаётся в **`backend/.gitignore`** (правила для `media/*` и `media/**/*` с исключениями для `.gitkeep`). Сами папки при этом остаются в репозитории.

### Улучшение UX при логине/регистрации

Чтобы избежать перезагрузки страницы и сделать ошибки более понятными:

- В `lib/api.ts`:
  - 401‑обработка **не перенаправляет** на `/login` для путей `/api/auth/login` и `/api/auth/register`
  - Логин/регистрация обрабатываются полностью на клиенте без полной перезагрузки

- В `(auth)/login` и `(auth)/register`:
  - Добавлены локальные состояния `formError`
  - Сообщения об ошибках (`err.message` из API) отображаются:
    - в toast‑уведомлении
    - отдельной строкой под формой (улучшенные и понятные подсказки)
  - Для пароля и логина добавлены текстовые подсказки по формату

### Запуск проекта локально

#### 1. Подготовка окружения

В корне проекта лежит файл `.env` (пример):

- Параметры Django:
  - `SECRET_KEY` — секретный ключ Django
  - `DEBUG` — режим отладки (`True/False`)
- Параметры базы:
  - `DB_NAME`, `DB_USER`, `DB_PASSWORD`, `DB_HOST`, `DB_PORT`
- Параметры Docker‑окружения:
  - `SERVER_IP` — хост, который будет использоваться во фронтенде для API (например, `localhost` или внешний IP сервера)

Убедитесь, что значения в `.env` корректны для вашей среды.

#### 2. Запуск через Docker

В корне репозитория:

```bash
docker-compose up --build
```

Будут подняты сервисы:

- `db` — PostgreSQL
- `backend` — Django + Gunicorn (`/api/...`)
- `frontend` — Next.js (SSR/SPA)
- `nginx` — reverse‑proxy на 80‑м порту

После успешного запуска:

- фронтенд будет доступен по адресу `http://<SERVER_IP>/`
- backend‑API — по `http://<SERVER_IP>/api/`

#### 3. Локальный запуск без Docker (для разработки)

- **Backend**
  1. Перейти в `backend/`
  2. Установить зависимости (`pip install -r requirements.txt`)
  3. Применить миграции:

     ```bash
     python manage.py migrate
     ```

  4. Запустить dev‑сервер:

     ```bash
     python manage.py runserver
     ```

- **Frontend**
  1. Перейти в `frontend/`
  2. Установить зависимости:

     ```bash
     npm install
     ```

  3. Создать `.env.local` с переменной:

     ```bash
     NEXT_PUBLIC_API_URL=http://localhost:8000
     ```

  4. Запустить dev‑режим:

     ```bash
     npm run dev
     ```

### Основные сценарии использования

- **Регистрация и вход**
  - Пользователь регистрируется или входит, не теряя введённые данные при ошибках
  - При неуспешном входе/регистрации страница не перезагружается, ошибки отображаются явно

- **Работа с профилем**
  - Редактирование логина, email, имени, дат рождения/смерти
  - Валидация дат на фронте и бэке, отображение возраста

- **Семейные деревья**
  - Создание семей, добавление участников
  - Построение древа через `TreeNode`/`TreeEdge` с соблюдением правил генеалогии (связь только родитель → ребёнок, проверка дат и разницы возрастов; см. раздел «Генеалогическое древо»)
  - Редактирование узлов с фото, биографией и корректной логикой дат; при изменении даты рождения проверяется согласованность с уже созданными связями

- **Воспоминания и медиа**
  - Создание воспоминаний, привязка к людям и участникам
  - Загрузка фото/видео, просмотр и редактирование
